# Code of Conduct

AtomVM CBOR is a small technical project. We want it to stay friendly, useful, and safe for everyone who contributes, opens an issue, or reviews a change.

## Our standard

Please be kind, direct, and constructive.

Good participation includes:

- explaining technical concerns clearly
- assuming good intent first
- keeping discussions focused on code, documentation, tests, and releases
- giving enough context when reporting a bug
- being patient with contributors who use different tools, operating systems, or experience levels

Unacceptable participation includes:

- harassment, personal attacks, insults, or discriminatory language
- repeatedly derailing technical discussions
- publishing private information or private messages
- sharing sensitive credentials or private infrastructure details
- pressuring maintainers to accept unsafe changes

## Public repository safety

This repository is public. Please do not open pull requests or issues that include private operational material.

Do not include:

- sensitive credentials
- private local paths
- private infrastructure details
- assistant-specific instruction files
- internal runtime state or private workflow notes

The repository CI rejects known internal or assistant-specific files so the public project stays clean.

## Reporting problems

If you see a behavior problem or public content that should not be public, please report it clearly and calmly in an issue or contact the maintainers directly.

When reporting a technical issue, include:

- what you expected to happen
- what actually happened
- the command you ran
- relevant versions, such as Erlang/OTP, AtomVM, ESP-IDF, Docker, container runtime, or operating system version

## Maintainer response

Maintainers may edit, close, hide, or remove content that violates these rules. Repeated or severe violations may lead to blocked participation.

The goal is not to be bureaucratic. The goal is to keep the project open, useful, and safe.
